场景 A：认知沉淀模型

Vibe Coding 场景下的软件认知沉淀机制研究

面向快速试错、代码经常被抛弃的原型项目，研究如何从 AI 对话和实验过程提取可复用的软件认知。

研究问题

在快速原型和 Vibe Coding 中，如何让被抛弃的代码不带走已经验证过的业务认知和工程判断？

摘要

探索型 AI 编程强调速度和反馈，代码经常被重写或删除，但实验目标、失败原因、用户反馈、有效 Prompt 和架构约束具有长期价值。本文提出面向 Vibe Coding 的软件认知沉淀机制，以实验日志和结构化提取卡为核心，把对话、Prompt、AI 输出、运行结果和用户反馈转化为可复用的用户故事、验收标准、领域概念、架构约束和风险假设。该机制的目标不是保存所有临时代码，而是保存可进入正式工程阶段的稳定认知。

关键词

Vibe Coding原型开发实验日志认知沉淀AI 对话治理

核心贡献

  • 提出探索型 AI 编程中“认知比临时代码更值得版本管理”的核心命题。
  • 设计实验日志与结构化提取卡两层文档模型，用于从 AI 对话中沉淀稳定认知。
  • 给出从实验记录到用户故事、验收标准和架构约束的转换流程。

引言

Vibe Coding 降低了从想法到可运行原型的距离。开发者可以通过连续对话让 AI 快速生成界面、接口和业务逻辑，并在用户反馈中不断调整方向。然而，探索型开发的代码生命周期很短，许多实现会被放弃。如果团队只管理代码版本，就会丢失大量已经验证过的认知。

本文认为，探索型 AI 编程真正需要版本管理的不是临时代码，而是实验认知。实验目标、失败原因、有效提示词、用户反应和约束判断，比某一次生成的代码更具长期价值。

问题分析

Vibe Coding 的知识流失主要表现为三类。第一，对话过程不可复现，后续成员很难理解某个原型为何这样设计。第二，失败实验没有结构化记录，导致团队重复尝试同类方案。第三，成功实验没有及时转化为需求和约束，进入正式开发时仍需要重新讨论。

这些问题说明，探索型项目需要一种低摩擦但结构化的文档模型，既不阻碍快速试错，又能把关键认知从对话中提取出来。

认知沉淀模型

本文提出“实验日志 + 结构化提取卡”的两层模型。实验日志记录每次试验的目标、初始假设、关键 Prompt、AI 输出、人工修改、运行结果、用户反馈、失败原因和结论。它保留过程证据，支持复盘。

结构化提取卡则从实验日志中抽取稳定认知，包括用户故事、验收标准、领域概念、架构约束、交互规则、风险假设和后续决策。提取卡不是流水账，而是可进入正式产品和工程文档的知识单元。

实施流程

第一步，在每次原型实验开始时写下目标和假设，避免对话漂移。第二步，在实验结束后用固定模板记录结果，特别是失败原因。第三步，对成功或有启发的实验生成提取卡。第四步，在阶段复盘时合并重复卡片，形成正式用户故事、验收标准和架构约束。第五步，将这些文档纳入后续项目的上下文，驱动 AI 生成更稳定的代码。

讨论

认知沉淀模型的难点在于控制记录成本。如果每次对话都要求详尽记录，会削弱探索速度。因此，该模型强调事件驱动记录：只有当实验验证了关键假设、暴露了重要风险、产生了用户反馈或形成了可复用设计判断时，才需要生成提取卡。

该模型也可以与 AI 自动摘要结合。AI 可以从对话和代码变更中生成初稿，但最终的认知判断仍应由人确认。

结论

Vibe Coding 场景下，代码生成速度不应掩盖知识管理问题。通过实验日志和结构化提取卡，团队可以在不牺牲探索速度的前提下，把 AI 对话中的有效认知转化为可复用的软件知识资产。

研究设计

可采用行动研究方法，在真实原型项目中引入实验日志和提取卡。对比引入前后团队重复试错次数、需求澄清时间和正式开发返工量。

也可设计受控实验，让两个小组完成同一原型任务，一组只保存代码和聊天记录，另一组使用认知沉淀模型，比较后续接手开发的理解效率和实现一致性。

评价设计

本文可采用案例研究、对比实验、专家评审或企业项目复盘进行验证。核心评价指标包括：

  • 实验日志完整率
  • 有效结论提取率
  • 重复试错减少量
  • 提取卡到用户故事转化率
  • 后续开发中的认知复用次数

局限与未来工作

  • 认知沉淀质量受到记录者判断能力影响，完全自动化摘要可能遗漏关键业务判断。
  • 探索型项目节奏很快，文档模板必须保持轻量，后续应研究最小字段集。
